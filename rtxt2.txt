Чек-лист: Перевод Django-проекта в "боевой режим"
Это базовый, но очень важный список шагов, которые нужно выполнить перед тем, как твой сайт увидят реальные пользователи.

1. DEBUG = False
Это самое главное правило безопасности.

Что делает: В photographer_project/settings.py измени DEBUG = True на DEBUG = False.

Зачем: Когда DEBUG включен, Django показывает подробные страницы с ошибками, которые могут содержать критически важную информацию о твоем проекте (настройки, пути к файлам и т.д.). В боевом режиме пользователи должны видеть только стандартную страницу ошибки (например, 404 или 500), а все детали должны писаться в лог-файлы на сервере.

2. SECRET_KEY
Что делает: Убедись, что ключ в settings.py — это длинная, случайная и уникальная строка. Никогда не выкладывай его в публичные репозитории (например, на GitHub)! В идеале, его нужно хранить в переменных окружения на сервере.

Зачем: Этот ключ используется для криптографической подписи сессий и CSRF-токенов. Если его украдут, злоумышленники смогут подделывать сессии пользователей.

3. ALLOWED_HOSTS
Что делает: В settings.py укажи доменное имя твоего сайта.

ALLOWED_HOSTS = ['your-domain.com', '[www.your-domain.com](https://www.your-domain.com)']

Зачем: Это защищает от атак с подменой HTTP-заголовка Host. * использовать в production категорически нельзя.

4. Настройка статических файлов
Что делает: Django не умеет эффективно раздавать статические файлы (CSS, JS, картинки) в боевом режиме. Для этого используется веб-сервер (например, Nginx).

Шаги:

В settings.py укажи STATIC_ROOT:

STATIC_ROOT = BASE_DIR / 'staticfiles'

Выполни команду python manage.py collectstatic. Она соберет все статические файлы из твоих приложений в одну папку staticfiles.

Настрой свой веб-сервер (Nginx) так, чтобы он раздавал файлы из этой папки по URL, указанному в STATIC_URL.

6. Настройка WSGI-сервера (Gunicorn)
Что делает: runserver — это однопоточный сервер для разработки. В production используется специальный WSGI-сервер, например, Gunicorn.

Как это работает: Gunicorn запускает несколько "рабочих" процессов твоего Django-приложения, которые могут обрабатывать запросы параллельно.

Пример команды для запуска: gunicorn photographer_project.wsgi:application --bind 0.0.0.0:8000

7. Настройка веб-сервера / Reverse Proxy (Nginx)
Что делает: Перед Gunicorn обычно ставят веб-сервер, такой как Nginx.

Зачем он нужен (его задачи):

Принимать все запросы с порта 80 (HTTP) и 443 (HTTPS).

Раздавать статические файлы (CSS, JS) напрямую, не нагружая Django.

Раздавать медиа-файлы (/media/) напрямую.

Перенаправлять (проксировать) все остальные запросы на Gunicorn, который уже выполняет код Django.

Настраивать HTTPS (SSL-сертификаты, например, от Let's Encrypt).

Упрощенная схема работы в Production:

(Интернет) ---> Nginx ---> Gunicorn ---> Django
                  |
                  +---> (отдает static/media файлы)

Это основы. Каждый пункт — это отдельная большая тема, но этот чек-лист даст тебе правильное направление для изучения и подготовки к запуску.
